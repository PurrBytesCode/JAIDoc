# JPA Embedding Converter

> Converts 384-dimensional `float[]` embedding arrays to/from `byte[]` BLOBs for SQLite storage.

## Context

SQLite has no native vector column type. To store the 384-dimensional embeddings generated by the multilingual-e5 ONNX
model, a JPA `AttributeConverter` serializes `float[]` to `byte[]` using `ByteBuffer/FloatBuffer`. This converter is
applied automatically by Hibernate when persisting or reading `JdkDocChunk.embedding`.

## Feature Inputs

- `src/main/java/com/purrbyte/ai/model/converter/FloatArrayConverter.java` — The converter implementation
- `src/test/java/com/purrbyte/ai/model/converter/FloatArrayConverterTest.java` — Tests for the converter
- `src/main/java/com/purrbyte/ai/domain/JdkDocChunk.java` — Entity that uses the converter (for context)

## Scope

In: float[] ↔ byte[] serialization logic, null handling, BLOB storage format
Out: The embedding model itself (covered in feature-embedding-service), Hibernate Search vector indexing (covered in
feature-data-models)

## Implementation

### Architecture

The converter is a plain JPA `AttributeConverter<float[], byte[]>` annotated with `@Converter`. JPA discovers it
automatically — no registration needed. It's applied to the `JdkDocChunk.embedding` field via
`@Convert(converter = FloatArrayConverter.class)`.

The conversion logic is straightforward:

- **To database**: `float[]` → `ByteBuffer.allocate(n * 4)` → `FloatBuffer.put()` → `byte[]`
- **From database**: `byte[]` → `ByteBuffer.wrap()` → `FloatBuffer.get()` → `float[]`

### Files

- `src/main/java/com/purrbyte/ai/model/converter/FloatArrayConverter.java` — The converter implementation

### Data Flow

```
JdkDocChunk.embedding (float[])
    │
    ├──→ FloatArrayConverter.convertToDatabaseColumn()
    │        └── ByteBuffer.allocate(384 * 4) → FloatBuffer.put() → byte[] (4096 bytes)
    │
    └──→ FloatArrayConverter.convertToEntityAttribute()
            └── ByteBuffer.wrap(byte[]) → FloatBuffer.get() → float[]
```

### Configuration

- `src/main/java/com/purrbyte/ai/domain/JdkDocChunk.java` — `@Convert(converter = FloatArrayConverter.class)` annotation
  on the embedding field

## Tests

- `src/test/java/com/purrbyte/ai/model/converter/FloatArrayConverterTest.java` — 4 test cases:
    - `roundTripsFloatArray()` — 4-element array round-trip
    - `convertsNullToNull()` — null input returns null both directions

    - `roundTripsFullDimension()` — full 384-dimension embedding round-trip

## Notes

- The converter is stateless — no instance variables, no thread safety concerns
- The `ByteBuffer.allocate()` path in `convertToDatabaseColumn` uses `buffer.array()` which returns the internal backing
  array of the ByteBuffer. This is safe because we're immediately returning it and the ByteBuffer goes out of scope.
- The `ByteBuffer.wrap()` path in `convertToEntityAttribute` does not copy the byte array — it creates a view. The
  returned float[] is a copy via `FloatBuffer.get(result)`, so the caller owns the float[] independently.
- The 384 dimension is hard-coded in the Hibernate Search `@VectorField(dimension = 384)` annotation on
  `JdkDocChunk.embedding`. If the model changes to a different dimension, both annotations must be updated together.
