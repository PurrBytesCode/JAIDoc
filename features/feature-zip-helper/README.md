# ZIP Helper

> Utility to find entries in ZIP files when entries may be under a version-prefixed directory.

## Context

When reading documentation ZIPs generated by the doclet, entries may be stored at different path depths depending on the
JDK version and how the ZIP was created. For example, `index.json` might be at the root (`index.json`), under a version
directory (`jdk-25.0.3/index.json`), or deeply nested (`jdk-25.0.3-ga/modules/java.base/index.json`). This utility
abstracts away the path discovery logic so callers don't need to know the exact path.

## Feature Inputs

- `src/main/java/com/purrbyte/ai/util/ZIPHelper.java` — The utility implementation
- `src/test/java/com/purrbyte/ai/util/ZIPHelperTest.java` — 8 test cases covering all edge cases

## Scope

In: ZIP entry lookup with version-prefixed path handling, Windows-style path support, deep directory traversal
Out: ZIP file creation/modification, ZIP extraction (covered in DocumentationService), ZIP compression

## Implementation

### Architecture

The utility is a single static method `findZipEntry(ZipFile, String name)` that iterates through all entries in the ZIP
and returns the first entry whose name matches one of three patterns:

1. **Exact match**: `entry.getName().equals(name)` — for entries at the root
2. **Version-prefixed match**: `entryName.endsWith("/" + name)` — for entries under a version directory (e.g.,
   `jdk-25.0.3/index.json`)
3. **Windows-style match**: `entryName.replace('\\', '/').endsWith("/" + name)` — for entries with backslash paths

### Files

- `src/main/java/com/purrbyte/ai/util/ZIPHelper.java` — The utility implementation

### Data Flow

```
ZIPHelper.findZipEntry(zipFile, "index.json")
    │
    ├──→ Iterate all ZIP entries
    │     │
    │     ├── Exact match: "index.json" == "index.json" ✓
    │     ├── Version-prefixed match: "jdk-25.0.3/index.json".endsWith("/index.json") ✓
    │     └── Windows-style match: "jdk-25.0.3\\index.json".replace('\\', '/').endsWith("/index.json") ✓
    │
    └──→ Return the first matching entry, or null if no match
```

### Configuration

None — the utility is stateless and requires no configuration.

## Tests

- `src/test/java/com/purrbyte/ai/util/ZIPHelperTest.java` — 8 test cases:
    - `exactMatch_returnsEntry()` — entry at root level
    - `versionPrefixedDirectory_returnsEntry()` — entry under version directory
    - `versionPrefixedWithSlash_returnsEntry()` — entry under non-version-prefixed directory
    - `nonExistentEntry_returnsNull()` — entry doesn't exist
    - `emptyZip_returnsNull()` — empty ZIP file
    - `noVersionPrefix_noMatch()` — entry exists but name doesn't match
    - `multipleVersionDirectories_returnsFirstMatch()` — multiple version directories, returns first match
    - `windowsStylePath_returnsEntry()` — entry with Windows backslash path
    - `deepVersionDirectory_returnsEntry()` — deeply nested entry

## Notes

- The method returns the **first** matching entry when multiple matches exist (e.g., `index.json` under both
  `jdk-21.0.11/` and `jdk-25.0.3/`). The order is determined by the ZIP entry enumeration order, which is not
  guaranteed.
- The `replace('\\', '/')` call normalizes Windows-style paths to Unix-style for consistent matching. This handles ZIPs
  created on Windows where the entry names use backslashes.
- Empty directory entries (e.g., `jdk-25.0.3/`) are included in the enumeration and checked against the name, but they
  won't match because they end with `/` not the target filename.
